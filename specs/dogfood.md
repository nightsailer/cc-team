# Dogfood: Replace Hook-based Context Guard with CCT Relay

## Context

Current context guard uses 5 loose hook files in `.claude/hooks/` + PID matching + pending/ queue + 3-state machine. Problems: complex, fragile, not distributable, requires manual `settings.local.json` editing.

**Goal**: (1) Unify context relay for team and standalone via CCT's relay mechanism. (2) Package hooks as Claude Code plugin — auto-registered via `--plugin-dir`, zero manual config. Distributable via `pip install cc-team`.

## Architecture

```
Distribution: Claude Code Plugin (plugin/ dir) → auto-registers hooks
Signal:       cct _hook statusline → usage.json
Trigger:      cct _hook stop → threshold → `cct relay` / `cct team relay --handoff`
Core:         _context_relay.py → graceful_exit + spawn new + inject handoff
Inject:       Agent=inbox prompt | TL/Standalone=tmux send_command after ready detection

User setup: claude --plugin-dir /path/to/cc-team/plugin
       or:  cct setup  (auto-configures plugin path)
```

### Key Design Decisions

1. **No `context-relay` command group** — extend existing `cct team relay` / `cct agent relay` with `--handoff <path>`. New top-level `cct relay` only for standalone mode.

2. **CCT Session ID** — persistent ID spanning multiple CC sessions:
   - Generated by `cct session start`, set as `CCT_SESSION_ID` env var
   - Survives relay: new CC process inherits same `CCT_SESSION_ID`
   - All relay artifacts stored under `.claude/relay/{cct_session_id}/`
   - Team mode: `CCT_SESSION_ID = {team_name}:{member_name}` or auto-generated
   - No `cct` = no relay support (no fallback to PID)

3. **Backend Protocol** — `AgentBackend` (types.py) already exists. Extend with relay methods (`graceful_exit`, `detect_ready`). TmuxBackend wraps existing TmuxManager. SubprocessBackend left as stub.

4. **Multi-session standalone** — each `cct session start` instance gets unique `CCT_SESSION_ID`, handoff paths never conflict.

```
CCT Session (persistent, one per cct session start)
├── CC Session #1 (context exhausted → relay)
├── CC Session #2 (context exhausted → relay)
└── CC Session #3 (current)

Storage: (default .claude/cct/, override via CCT_PROJECT_DATA_DIR)
.claude/cct/
├── config.json                        # CCT config (threshold, defaults)
└── relay/{cct_session_id}/
    ├── handoff.md                     # current handoff document
    ├── usage.json                     # statusline hook writes current usage
    └── history.json                   # CC session chain [sid1, sid2, sid3]
```

## Implementation

### Step 1: Create `src/cc_team/hooks/` package (hook handlers)

Move all hook logic from loose `.claude/hooks/*.py` into the package. Each module is invoked via `cct _hook <name>` (routed through the `cct` console_scripts entry point to ensure correct virtualenv).

```
src/cc_team/hooks/
├── __init__.py
├── _common.py          # shared: project_dir, read_json, write_json, session_paths, load_config
├── stop.py             # Stop hook: context relay trigger (2-phase)
└── statusline.py       # StatusLine hook: usage monitoring + bar rendering
```

Each module exposes `main()` + `if __name__ == "__main__": main()` for direct execution.

**`stop.py`** — simplified 2-phase (replaces 3-state machine):
- Phase 1 (no handoff): block, request handoff.md at `.claude/relay/{cct_session_id}/handoff.md`
- Phase 2 (handoff exists): launch relay in background (`cct relay --handoff ...` or `cct team relay --handoff ...`), return allow
- Read `CCT_SESSION_ID` from env to determine relay path + mode (standalone vs team)
- Config from `{project}/.claude/hooks/context-relay-config.json`:
  ```json
  {"threshold": 80, "max_block_count": 3, "team_name": ""}
  ```
- Skip subagents (check `agent_name` from usage.json)
- Determine backend_id via `tmux list-panes -F "#{pane_pid} #{pane_id}"` + PID matching (tmux backend only)

**`statusline.py`** — migrated from `.claude/hooks/statusline.py`:
- Write usage to `.claude/relay/{cct_session_id}/usage.json` (read `CCT_SESSION_ID` from env)
- Fallback: if no `CCT_SESSION_ID`, use `session_id` as key (read-only status bar, no relay support)

**`_common.py`** — migrated from `_cg_common.py`:
- `project_dir()`, `read_json()`, `write_json()`, `atomic_write_json()`, `load_config()`
- `cct_data_dir()` — `CCT_PROJECT_DATA_DIR` env var, fallback `{project}/.claude/cct/`
- `relay_paths(cct_session_id) -> dict` — returns `{dir, handoff, usage, history}`, 基于 `cct_data_dir()`
- Remove `get_host_pid()` (no PID-based fallback)
- Remove `session_paths()` (replaced by `relay_paths()`)
- Remove `SESSIONS_DIR`, `PENDING_DIR` constants

### Step 2: Create Claude Code Plugin structure

Create `plugin/` directory in repo root:

```
plugin/
├── .claude-plugin/
│   └── plugin.json              # Plugin manifest
└── hooks/
    └── hooks.json               # Auto-registers Stop + StatusLine hooks
```

**`plugin/.claude-plugin/plugin.json`**:
```json
{
  "name": "cc-team",
  "description": "Claude Code multi-agent team orchestration - context relay hooks",
  "version": "0.1.0",
  "author": {"name": "cc-team"}
}
```

**`plugin/hooks/hooks.json`**:
```json
{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "cct _hook stop",
            "timeout": 30
          }
        ]
      }
    ]
  },
  "statusLine": {
    "type": "command",
    "command": "cct _hook statusline",
    "padding": 2
  }
}
```

**Hook 环境策略**：
- hooks.json 命令统一使用 `cct _hook <name>`，通过 console_scripts 入口 `cct` 分发，确保在正确的 Python 环境（uv tool / pipx virtualenv）中运行
- **架构防线**：hooks 层始终保持尽可能薄（sensor/trigger），重活委托给 `cct` CLI 命令（如 `cct relay --handoff ...`），`cct` 作为 console_scripts 安装，环境天然正确

**Usage**: `claude --plugin-dir /path/to/cc-team/plugin`

### Step 2b: Add `cct setup` command

Add to CLI a `setup` subcommand:

```
cct setup              # Print plugin-dir path + usage instructions
cct setup --install    # Symlink plugin to ~/.claude/plugins/cc-team (auto-load)
```

### Step 2c: Add `cct session start` command

```
cct session start [<claude args...>]    # Generate CCT_SESSION_ID, launch claude
```

`cct session start` 之后的所有参数直接透传给 `claude` CLI，例如：
```
cct session start --model sonnet --resume
cct session start --plugin-dir /path/to/plugin --allowedTools 'Bash(git:*)'
```

Implementation:
- Generate UUID → set `CCT_SESSION_ID` env var
- Write `.claude/relay/{cct_session_id}/history.json` with initial entry
- `exec claude <passthrough args...>` (inherits env) or `tmux split-window` + send command
- 使用 `parse_known_args`：`cct session start` 不定义自有参数，所有未识别参数透传给 claude

### Step 3: Extend `AgentBackend` Protocol + create `_context_relay.py`

**Extend `AgentBackend`** in `types.py` with relay methods:
```python
async def graceful_exit(self, backend_id: str, timeout: int) -> None:
    """Send /exit and wait for process to die."""
    ...

async def detect_ready(self, backend_id: str, timeout: int) -> bool:
    """Wait until session is ready to accept input."""
    ...
```

**`ProcessManager`** (TmuxBackend): implement using existing `_graceful_exit_pane` logic + `TmuxManager.detect_state()`.

**`SubprocessBackend`**: stub implementation, raise `NotImplementedError`.

**`src/cc_team/_context_relay.py`** — core relay logic:

```python
@dataclass
class RelayRequest:
    cct_session_id: str        # CCT session ID
    handoff_path: str          # path to handoff.md
    model: str = DEFAULT_MODEL
    timeout: int = 30
    cwd: str = ""

@dataclass
class RelayResult:
    old_backend_id: str | None
    new_backend_id: str
    cct_session_id: str
    handoff_injected: bool = False
```

**Shared helpers** (reuse existing):
- `_read_handoff(path) -> str` — read handoff file
- `_format_handoff_prompt(content) -> str` — wrap in relay context header
- `_update_history(cct_session_id, new_cc_session_id)` — append to history.json
- `_inject_handoff(backend, backend_id, content)` — detect_ready + send_input

**Entry points**:

1. `relay_standalone(request, *, backend: AgentBackend, backend_id: str) -> RelayResult`
   - `backend.graceful_exit(backend_id)`
   - Spawn new CC session (split_window + send command)
   - `_inject_handoff(backend, new_id, content)`
   - `_update_history(cct_session_id, new_cc_session_id)`

2. `relay_lead(request, *, team_name: str) -> RelayResult`
   - Find TL from config, `backend.graceful_exit`
   - `TeamManager.rotate_session()`
   - `ProcessManager.spawn_lead()`
   - `_inject_handoff(backend, new_bid, content)`
   - `sync_member_states()`
   - `_update_history()`

3. `relay_agent(request, *, team_name: str, agent_name: str) -> RelayResult`
   - `backend.graceful_exit`, `mgr.remove_member()`
   - `spawn_agent_workflow()` — handoff content as prompt → inbox
   - `_update_history()`

### Step 4: Extend CLI commands

**`cct relay`** (new top-level, standalone only):
```
cct relay --handoff <path> [--backend-id %42] [--model MODEL] [--timeout 30]
```
- Reads `CCT_SESSION_ID` from env (required, error if not set)
- Calls `relay_standalone()`

**`cct team relay`** (extend existing):
```
cct team relay [--handoff <path>] [--model MODEL] [--timeout 30]
```
- `--handoff` is optional; when provided, inject handoff after spawning new TL

**`cct agent relay`** (extend existing):
```
cct agent relay --name X [--handoff <path>] [--prompt "..."] [--timeout 30]
```
- `--handoff` is optional; when provided, use handoff content as the respawn prompt

**`cct session start`** (new):
```
cct session start [<claude args...>]
```

**`cct setup`** (new):
```
cct setup [--install]
```

## Open Questions

1. **Plugin hooks.json format for StatusLine**: Need to verify if `statusLine` can be defined inside `hooks.json` alongside event hooks, or if it requires separate configuration. If not supported, `statusLine` may still need `settings.local.json` entry.

2. **`cct session start` and tmux**: Should `cct session start` always use tmux (split-window)? Or should it work without tmux (just exec claude in current terminal)? Current recommendation: exec in current terminal by default, `--tmux` flag to use split-window.
